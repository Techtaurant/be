# Spring REST Docs + OpenAPI 통합 가이드

테스트, OpenAPI 스펙, Swagger UI를 어떻게 조합할지 프레임워크별로 비교하고, Spring Boot에서는 어떤 식으로 구현하면 좋은지 정리한다.

## 개요

이 문서는 원래 Spring REST Docs와 Swagger UI를 함께 쓰는 방법을 설명하기 위해 작성했다. 여기에 한 단계 더 나아가, 같은 요구를 **FastAPI**나 **NestJS**에서도 구현할 수 있는지까지 함께 정리한다.

질문을 먼저 다시 정리하면 아래와 같다.

- 테스트가 실제 API 동작을 검증해야 한다.
- OpenAPI 스펙도 함께 유지되어야 한다.
- Swagger UI 같은 인터랙티브 문서 화면도 제공하고 싶다.
- 가능하면 문서와 구현이 쉽게 어긋나지 않아야 한다.

핵심 결론은 단순하다. **비슷한 결과물은 세 프레임워크 모두 만들 수 있지만, 문서를 생성하는 방식은 다르다.** Spring REST Docs는 테스트가 문서를 만드는 쪽이고, FastAPI와 NestJS는 코드가 OpenAPI를 만들고 테스트가 그 결과를 검증하는 쪽에 가깝다.

---

## 결론 먼저: 프레임워크별 가능 여부

| 프레임워크 | 가능한가 | 문서의 원천 | Swagger UI | Spring REST Docs와 같은가 |
|------------|----------|-------------|------------|----------------------------|
| Spring Boot | 가능 | 테스트 실행 결과 + `restdocs-api-spec` | 추가 설정 필요 | 예, 가장 가깝다 |
| FastAPI | 가능 | 타입 힌트 + Pydantic 모델 + 라우트 정의 | 기본 제공 (`/docs`) | 아니오 |
| NestJS | 가능 | 데코레이터 + 런타임 메타데이터 + CLI Plugin | 기본 제공 (`SwaggerModule.setup`) | 아니오 |

조금 더 정확히 표현하면 다음과 같다.

- **Spring Boot**는 `Spring REST Docs -> OpenAPI 생성 -> Swagger UI 서빙` 흐름이 가능하다.
- **FastAPI**는 `코드에서 OpenAPI 자동 생성 -> Swagger UI 자동 제공 -> 테스트로 스키마/동작 검증` 흐름이 가능하다.
- **NestJS**는 `@nestjs/swagger로 OpenAPI 생성 -> Swagger UI 제공 -> E2E 테스트로 스키마 검증` 흐름이 가능하다.

즉, **셋 다 “문서 + 스키마 + 테스트” 조합은 가능하지만, Spring만 “테스트가 문서를 직접 생성한다”는 철학을 공식적으로 밀어준다.**

> 출처: [Spring REST Docs](https://docs.spring.io/spring-restdocs/docs/current/reference/htmlsingle/), [restdocs-api-spec](https://github.com/ePages-de/restdocs-api-spec), [FastAPI](https://fastapi.tiangolo.com/), [NestJS](https://docs.nestjs.com/openapi/introduction)

## 문서 생성 방식을 먼저 구분하기

프레임워크를 비교할 때는 먼저 **문서가 어디서 만들어지는지**를 구분하는 게 중요하다.

### 테스트 주도 문서화

이 방식은 테스트가 실제 HTTP 요청/응답을 수행하고, 그 결과를 기반으로 문서 조각이나 스펙을 만든다.

- 장점은 문서와 실제 동작이 어긋나기 어렵다는 점이다.
- 단점은 테스트 인프라와 문서 생성 파이프라인을 따로 구성해야 한다는 점이다.
- Spring REST Docs가 대표적이다.

### 코드 주도 OpenAPI 생성

이 방식은 컨트롤러, 데코레이터, 타입 힌트, DTO, 모델 정의에서 OpenAPI를 만든다.

- 장점은 설정이 단순하고 Swagger UI까지 빠르게 제공할 수 있다는 점이다.
- 단점은 테스트가 문서를 직접 만들지 않기 때문에, 문서 정확성을 보장하려면 별도의 검증 전략이 필요하다는 점이다.
- FastAPI와 NestJS가 여기에 가깝다.

### 실무에서 자주 쓰는 보완 방식

FastAPI와 NestJS는 Spring REST Docs처럼 테스트가 문서를 생성하지는 않지만, 아래 방식으로 충분히 강한 계약 검증을 만들 수 있다.

- 테스트에서 `/openapi.json` 또는 생성된 스키마를 검증한다.
- CI에서 OpenAPI 스펙 diff를 검사한다.
- schema validator 또는 contract testing 도구를 붙여 응답이 스펙과 일치하는지 확인한다.

이 차이를 먼저 이해하면, 왜 Spring에서는 REST Docs가 자연스럽고 FastAPI/NestJS에서는 Swagger 모듈 + 테스트 검증 조합이 자연스러운지도 같이 이해된다.

## Spring Boot에서 Spring REST Docs + Swagger 조합이 맞는 이유

Spring Boot에서는 테스트 주도 문서화라는 요구에 가장 잘 맞는 조합이 **Spring REST Docs + restdocs-api-spec + Swagger UI**다.

흐름은 아래처럼 정리할 수 있다.

1. MockMvc 테스트에서 요청과 응답을 검증한다.
2. 테스트 실행 중 문서 스니펫과 `resource.json`이 생성된다.
3. `openapi3` Gradle 태스크가 이를 모아 OpenAPI 스펙을 만든다.
4. 생성된 스펙을 Swagger UI에서 보여준다.

즉, **테스트 성공이 문서 생성의 전제조건**이 된다. 이 점이 Springdoc-openapi와 가장 큰 차이다.

> 출처: [Spring REST Docs](https://docs.spring.io/spring-restdocs/docs/current/reference/htmlsingle/), [restdocs-api-spec](https://github.com/ePages-de/restdocs-api-spec), [Creating API Documentation with Restdocs](https://spring.io/guides/gs/testing-restdocs/)

### springdoc-openapi와 비교

| 항목 | springdoc-openapi | Spring REST Docs + restdocs-api-spec |
|------|-------------------|--------------------------------------|
| 문서 생성 시점 | 런타임 | 테스트/빌드 타임 |
| 문서의 원천 | 어노테이션, DTO, 런타임 메타데이터 | 테스트 코드 |
| Swagger UI | 거의 즉시 사용 가능 | 추가 설정 필요 |
| 문서 정확성 보장 | 별도 검증 필요 | 테스트가 통과해야 생성 |
| 코드 침습성 | Swagger 어노테이션이 프로덕션 코드에 들어감 | 주로 테스트 코드에 집중 |

Spring에서 “문서와 실제 동작이 절대 어긋나면 안 된다”는 요구가 강하면, Spring REST Docs 쪽이 더 적합하다.

> 출처: [Spring REST Docs vs OpenAPI](https://www.baeldung.com/spring-rest-docs-vs-openapi)

## Spring Boot 구현 가이드

### 버전 호환성

| Spring Boot | restdocs-api-spec |
|-------------|-------------------|
| 2.x | 0.16.4 이하 |
| 3.x | 0.17.1 ~ 0.19.4 |
| 4.x | 0.20.x 이상 |

`restdocs-api-spec`의 최신 안정 버전은 2025년 6월 기준 `0.19.5`다. 다만 Spring Boot 3.x 프로젝트라면 README의 호환성 표를 먼저 보고 버전을 고르는 것이 안전하다.

> 출처: [restdocs-api-spec](https://github.com/ePages-de/restdocs-api-spec), [com.epages.restdocs-api-spec](https://plugins.gradle.org/plugin/com.epages.restdocs-api-spec)

```kotlin
}
```

### 테스트 작성 패턴

핵심은 `MockMvcRestDocumentationWrapper.document()`와 `ResourceSnippetParameters.builder()`를 함께 사용하는 것이다.

```kotlin
import com.epages.restdocs.apispec.MockMvcRestDocumentationWrapper
import com.epages.restdocs.apispec.ResourceDocumentation
import com.epages.restdocs.apispec.ResourceSnippetParameters
import org.springframework.restdocs.mockmvc.RestDocumentationRequestBuilders
import org.springframework.restdocs.payload.PayloadDocumentation.fieldWithPath
import org.springframework.restdocs.request.RequestDocumentation.parameterWithName

@Test
fun `게시글 단건 조회 API`() {
    mockMvc.perform(
        RestDocumentationRequestBuilders.get("/api/v1/posts/{id}", 1)
            .accept(MediaType.APPLICATION_JSON)
    )
        .andExpect(status().isOk)
        .andDo(
            MockMvcRestDocumentationWrapper.document(
                "post-get-detail",
                ResourceDocumentation.resource(
                    ResourceSnippetParameters.builder()
                        .summary("게시글 단건 조회")
                        .description("ID로 게시글의 상세 정보를 조회한다")
                        .tag("Post")
                        .pathParameters(
                            parameterWithName("id").description("게시글 ID")
                        )
                        .responseFields(
                            fieldWithPath("status").description("HTTP 상태 코드"),
                            fieldWithPath("data.id").description("게시글 ID"),
                            fieldWithPath("data.title").description("게시글 제목"),
                            fieldWithPath("data.content").description("게시글 본문"),
                            fieldWithPath("message").description("응답 메시지")
                        )
                        .build()
                )
            )
        )
}
```

여기서 중요한 점은 `RestDocumentationRequestBuilders`를 써야 path parameter 템플릿이 유지된다는 것이다. 그렇지 않으면 `/posts/{id}` 정보가 OpenAPI로 제대로 올라가지 않는다.

> 출처: [Spring REST Docs](https://docs.spring.io/spring-restdocs/docs/current/reference/htmlsingle/), [restdocs-api-spec](https://github.com/ePages-de/restdocs-api-spec)

### Bean Validation과 에러 응답 문서화

`ConstrainedFields`를 사용하면 DTO의 Bean Validation 제약조건을 OpenAPI 스키마에 반영할 수 있다. 또한 에러 응답은 별도 테스트 메서드로 작성해두면 같은 엔드포인트에 대해 여러 응답 코드가 스펙에 병합된다.

```kotlin
import com.epages.restdocs.apispec.ConstrainedFields

data class PostCreateRequest(
    @field:NotBlank
    @field:Length(max = 100)
    val title: String,

    @field:NotBlank
    val content: String,

    @field:NotNull
    val categoryId: Long,
)

@Test
fun `게시글 생성 API - 유효성 검증 실패`() {
    val invalidRequest = """{"title": "", "content": ""}"""

    mockMvc.perform(
        RestDocumentationRequestBuilders.post("/api/v1/posts")
            .contentType(MediaType.APPLICATION_JSON)
            .content(invalidRequest)
    )
        .andExpect(status().isBadRequest)
        .andDo(
            MockMvcRestDocumentationWrapper.document(
                "post-create-validation-error",
                ResourceDocumentation.resource(
                    ResourceSnippetParameters.builder()
                        .summary("게시글 생성")
                        .tag("Post")
                        .responseFields(
                            fieldWithPath("status").description("HTTP 상태 코드"),
                            fieldWithPath("data.errors").description("필드별 에러 메시지"),
                            fieldWithPath("message").description("에러 메시지")
                        )
                        .build()
                )
            )
        )
}
```

이 패턴은 이 프로젝트처럼 에러 응답 케이스를 꼼꼼히 문서화해야 하는 경우 특히 잘 맞는다.

> 출처: [restdocs-api-spec](https://github.com/ePages-de/restdocs-api-spec), [Spring REST Docs](https://docs.spring.io/spring-restdocs/docs/current/reference/htmlsingle/)

### OpenAPI 생성과 Swagger UI 서빙

테스트가 준비되면 아래 순서로 OpenAPI 스펙을 만들 수 있다.

```bash
./gradlew test openapi3
```

생성 경로는 `build/api-spec/openapi3.yaml`이다.

이 스펙을 Swagger UI에 붙이는 방법은 크게 세 가지다.

- `swagger-generator` 계열 Gradle 플러그인으로 정적 UI를 생성해 서빙한다.
- 기존 `springdoc-openapi`의 Swagger UI가 생성된 yaml을 읽게 만든다.
- ReDoc 같은 대체 UI로 정적 스펙을 노출한다.

```yaml
springdoc:
  swagger-ui:
    url: /docs/openapi3.yaml
```

> 출처: [Generate Swagger UI from Spring REST Docs](https://jdriven.com/blog/2021/10/Generate-Swagger-UI-from-Spring-REST-Docs)

## 다른 프레임워크에서도 가능한가

여기서부터가 이번에 추가된 핵심이다. 질문은 “Spring REST Docs + Swagger 같은 조합을 FastAPI, NestJS에서도 만들 수 있느냐”였고, 답은 **부분적으로 예**다.

다만 “같은 결과를 낸다”와 “같은 방식으로 동작한다”는 전혀 다른 말이다.

### FastAPI

FastAPI는 OpenAPI와 Swagger UI가 프레임워크에 기본 포함되어 있다.

- OpenAPI 스펙은 라우트, 타입 힌트, Pydantic 모델에서 자동 생성된다.
- Swagger UI는 기본적으로 `/docs`에, ReDoc은 `/redoc`에 제공된다.
- raw 스펙은 `/openapi.json`에서 확인할 수 있다.

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI(title="My API", version="1.0.0")

class User(BaseModel):
    id: int
    name: str

@app.get("/users/{user_id}", response_model=User, tags=["users"])
async def get_user(user_id: int):
    return {"id": user_id, "name": "Alice"}
```

테스트는 보통 `TestClient`로 작성한다.

```python
from fastapi.testclient import TestClient

client = TestClient(app)

def test_get_user():
    response = client.get("/users/1")
    assert response.status_code == 200
    assert response.json() == {"id": 1, "name": "Alice"}
```

즉, FastAPI에서는 **테스트가 문서를 생성하지 않는다.** 대신 코드가 OpenAPI를 만들고, 테스트가 그 동작과 스펙을 검증한다.

필요하면 `get_openapi()`를 이용해 스키마를 커스터마이징할 수도 있다.

```python
from fastapi.openapi.utils import get_openapi

def custom_openapi():
    if app.openapi_schema:
        return app.openapi_schema
    openapi_schema = get_openapi(
        title=app.title,
        version=app.version,
        description=app.description,
        routes=app.routes,
    )
    app.openapi_schema = openapi_schema
    return app.openapi_schema

app.openapi = custom_openapi
```

FastAPI에서 Spring REST Docs에 가장 가까운 운영 방식은 아래 쪽이다.

- 애플리케이션 코드가 `/openapi.json`을 만든다.
- 테스트가 `/openapi.json` 또는 실제 응답을 검증한다.
- 필요하면 Schemathesis 같은 도구로 OpenAPI 기반 contract test를 추가한다.

하지만 이것은 **docs-from-tests**가 아니라 **tests-against-docs**에 더 가깝다.

> 출처: [FastAPI](https://fastapi.tiangolo.com/), [Testing - FastAPI](https://fastapi.tiangolo.com/tutorial/testing/), [Extending OpenAPI - FastAPI](https://fastapi.tiangolo.com/how-to/extending-openapi/), [Schemathesis](https://schemathesis.readthedocs.io/)

### NestJS

NestJS는 `@nestjs/swagger`를 공식적으로 제공한다.

- 컨트롤러와 DTO에 붙은 데코레이터를 기반으로 OpenAPI를 생성한다.
- `SwaggerModule.createDocument()`가 애플리케이션 메타데이터를 읽어 스키마를 만든다.
- `SwaggerModule.setup()`으로 Swagger UI를 바로 띄울 수 있다.

```typescript
import { NestFactory } from '@nestjs/core';
import { SwaggerModule, DocumentBuilder } from '@nestjs/swagger';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  const config = new DocumentBuilder()
    .setTitle('Cats API')
    .setDescription('The cats API description')
    .setVersion('1.0')
    .addTag('cats')
    .build();

  const document = SwaggerModule.createDocument(app, config);
  SwaggerModule.setup('api', app, document);

  await app.listen(3000);
}
bootstrap();
```

테스트 관점에서는 E2E에서 생성된 스키마를 검증하는 패턴을 많이 쓴다.

```typescript
it('should produce a valid OpenAPI 3.0 schema', async () => {
  const document = SwaggerModule.createDocument(app, builder.build());
  const doc = JSON.stringify(document, null, 2);
  writeFileSync(join(__dirname, 'api-spec.json'), doc);
  await SwaggerParser.validate(document as any);
});
```

또한 CLI Plugin을 사용하면 DTO와 validator 정보를 보고 Swagger 메타데이터를 자동 보강할 수 있다.

```json
{
  "compilerOptions": {
    "plugins": [
      {
        "name": "@nestjs/swagger",
        "options": {
          "classValidatorShim": true,
          "introspectComments": true
        }
      }
    ]
  }
}
```

NestJS도 결국은 **코드가 문서를 만들고, 테스트가 그 결과를 검증하는 구조**다. Spring REST Docs처럼 테스트에서 요청/응답을 캡처해 문서 스니펫을 쌓는 공식 프레임워크는 없다.

> 출처: [NestJS](https://docs.nestjs.com/openapi/introduction), [NestJS](https://docs.nestjs.com/openapi/cli-plugin), [nestjs/swagger](https://github.com/nestjs/swagger)

### 공통 결론

세 프레임워크를 다시 한 줄로 정리하면 이렇다.

- **Spring REST Docs**: 테스트가 문서를 만든다.
- **FastAPI**: 코드가 문서를 만들고 테스트가 검증한다.
- **NestJS**: 데코레이터와 메타데이터가 문서를 만들고 테스트가 검증한다.

그래서 “Spring REST Docs + Swagger 같은 걸 FastAPI/NestJS에서도 하고 싶다”는 요청은, 실제로는 아래처럼 번역하는 것이 정확하다.

- FastAPI/NestJS에서도 **OpenAPI + Swagger UI + 테스트 검증 조합은 충분히 가능하다.**
- 하지만 **Spring REST Docs와 완전히 같은 흐름은 아니다.**
- 같은 문제를 다른 방식으로 푼다고 보는 편이 맞다.

---

## 어떤 조합을 선택하면 좋은가

### Spring REST Docs + restdocs-api-spec가 더 맞는 경우

- 테스트 결과가 곧 문서여야 하는 경우
- Swagger 어노테이션을 프로덕션 코드에 많이 넣고 싶지 않은 경우
- 에러 응답과 validation 규칙을 테스트 단위로 강하게 고정하고 싶은 경우
- API 문서 품질이 매우 중요한 경우

### FastAPI 기본 OpenAPI + 테스트 검증이 더 맞는 경우

- 빠르게 API와 Swagger UI를 제공해야 하는 경우
- Pydantic 기반 모델 정의만으로 문서를 자동 생성하고 싶은 경우
- OpenAPI는 자동 생성하되, 테스트는 별도로 관리해도 괜찮은 경우
- Schemathesis 같은 contract testing 도구를 붙일 계획이 있는 경우

### NestJS + @nestjs/swagger + E2E 검증이 더 맞는 경우

- TypeScript와 데코레이터 중심 개발 방식이 익숙한 경우
- DTO, class-validator, Swagger 메타데이터를 한 흐름으로 관리하고 싶은 경우
- 생성된 OpenAPI를 CI에서 검증하거나 SDK 생성에 활용하려는 경우

### 최종 정리

문서의 정확성을 최우선으로 보고 **테스트가 문서를 직접 만들어야 한다면 Spring REST Docs**가 가장 적합하다.

반대로 개발 속도와 기본 제공 Swagger UI가 더 중요하다면, **FastAPI와 NestJS의 기본 OpenAPI 생성 방식**이 더 자연스럽다. 이 경우 테스트의 역할은 문서 생성이 아니라 **계약 검증**으로 두는 것이 좋다.
